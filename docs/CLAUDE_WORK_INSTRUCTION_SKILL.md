# 실장님 전용 클로드/개발 지시문 작성 스킬

## 목적
이 문서는 아르바이트 급여·연차 관리 프로그램 및 기타 개발/운영 프로젝트 진행 시,
클로드 또는 유사한 개발 보조 AI에게 누락 없이, 안정적으로, 기존 정상 기능을 최대한 건드리지 않는 방식으로 작업을 지시하기 위한 고정 작성 기준입니다.

짧고 압축된 지시문보다,
현재 상태 / 누적 확정 구조 / 이번 작업 목적 / 금지 범위 / 수정 방식 / 회귀 체크 / 검증 / 보고 형식까지 포함된
길고 자세한 업무용 지시문이 결과물 안정성이 높고 누락이 적으므로,
항상 아래 원칙을 우선 적용합니다.

---

## 1. 말투 / 톤
- 항상 존댓말의 업무용 톤으로 작성합니다.
- 가볍거나 대화체가 아니라, 실제 실무 지시문처럼 작성합니다.
- 작업자가 바로 이해하고 실행할 수 있도록 명확하게 씁니다.
- 애매한 표현보다 판단 기준이 포함된 문장으로 작성합니다.

## 2. 길이 / 밀도
- 가능한 한 짧게 줄이지 않습니다.
- 복잡한 작업일수록 맥락을 충분히 넣습니다.
- 길고 자세한 지시문을 우선합니다.
- 누락 방지가 더 중요하므로, 이미 확인된 내용도 필요한 경우 다시 적습니다.

## 3. 출력 형태
- 항상 복붙 가능한 완성형 지시문으로 작성합니다.
- 설명형 답변보다 바로 클로드에 넣을 수 있는 원문 형태를 우선합니다.
- 필요 시 아래 2종을 함께 제공합니다.
  1. 클로드/개발용 작업 지시문
  2. 새 대화창 시작용 프롬프트

---

## 4. 클로드 지시문 작성 기준

클로드 지시문은 아래 6개 항목을 반드시 포함합니다.

### 4-1. 현재 상태와 완료된 작업 먼저 정리
지시문 시작부에 반드시 아래를 먼저 적습니다.
- 프로젝트명
- 현재 기준 브랜치 / 배포 상태 / 머지 상태
- 직전까지 완료된 핵심 작업
- 직전 작업에서 확인된 문제 원인과 해결 방향
- 지금 시점 기준으로 이미 확정된 사실

즉, 작업자가 지금 무엇이 끝난 상태인지부터 먼저 파악할 수 있어야 합니다.

### 4-2. 이번 작업의 목적과 우선순위 명확히 기재
반드시 아래를 구분해서 적습니다.
- 이번 작업의 최종 목적
- 이번 턴에서 꼭 끝내야 하는 1순위 작업
- 그 다음으로 진행할 2순위 / 3순위 작업
- 이번 턴에서 하지 말아야 할 확장 작업

목적은 모호하게 쓰지 말고,
행동 단위로 명확하게 적습니다.

예:
- 로그인 이슈 원인 분석
- leave_balance 원복 확인
- 브랜치 정리만 진행

### 4-3. 건드리면 안 되는 로직 / 파일 / 구조 분리 명시
반드시 별도 항목으로 분리해서 적습니다.

예시:
- firestore.rules
- functions/index.js
- AuthContext / App 라우팅
- roleGroup 매핑 구조
- 승인 단계 로직
- status 확장 체계
- delegation 구조
- 이미 정상 동작 중인 UI 흐름
- 배포 완료된 기능 전체

특히 아래 상황에서는 더 강하게 명시합니다.
- 이번 작업이 데이터 확인 / 브랜치 정리 / 로그 확인 성격일 때
- 작은 수정만 해야 할 때
- 과거에 연쇄 수정으로 정상 기능까지 깨진 경험이 있을 때

### 4-4. 수정 파일과 수정 방식 구체화
수정이 필요한 경우 아래를 반드시 씁니다.
- 어떤 파일을 수정해야 하는지
- 왜 그 파일을 수정하는지
- 어느 수준까지만 수정해야 하는지
- 어떤 방식으로 수정해야 하는지
- 가능하면 어떤 방식은 피해야 하는지

좋은 예:
- LeaveApprovalInbox.jsx에서 브라우저 기본 confirm을 제거하고 기존 ConfirmModal 재사용 방식으로만 수정해 주십시오.
- roleUtils.normalizeProfile()를 기준으로 하위호환만 보완하고, App.jsx 라우팅 구조 전체를 재설계하지 마십시오.

좋지 않은 예:
- 관련 파일들 적당히 수정
- 구조를 정리
- 필요한 부분 다 손봐라

### 4-5. 회귀 체크 포인트 포함
작업 후 반드시 확인해야 하는 항목을 적습니다.

예시:
- employee 로그인 진입 정상 여부
- manager 로그인 진입 정상 여부
- 상태 라벨 raw 값 노출 여부
- 승인 버튼 클릭 시 ConfirmModal 정상 호출 여부
- 반려 데이터 메타데이터 표시 정상 여부
- leave_balance 수치 일관성 여부
- working tree clean 여부
- 배포 후 실제 사용자 계정 테스트 여부

체크 포인트는 이번 수정 범위와 직접 연결된 것 위주로 적고,
필요하면 필수 확인과 가능하면 확인으로 구분합니다.

#### 추가 검증 원칙
- 코드, 데이터, 설정 변경이 있었다면 작업 마무리 후 반드시 버그 / 오류 / 회귀 테스트를 진행합니다.
- 테스트는 이번 수정 범위와 직접 연결된 핵심 기능부터 우선 확인합니다.
- 테스트를 실제로 실행한 경우에는 무엇을 어떻게 확인했는지 보고에 남깁니다.
- 테스트를 실행하지 못한 경우에는 `미실행`으로 명시하고, 미실행 사유와 다음 확인 필요 항목을 함께 남깁니다.
- 문서 작성이나 운영 정리처럼 실제 기능 변경이 없는 작업은 코드 테스트 대신 검토 완료 여부를 기록합니다.

### 4-6. 작업 후 보고 형식까지 지정
항상 마지막에 아래처럼 보고 형식을 지정합니다.

반드시 포함할 것:
- 무엇을 확인했는지
- 무엇을 수정했는지
- 어떤 파일을 건드렸는지
- 실제 결과가 어땠는지
- 미완료 항목이 있는지
- 다음 작업으로 이어질 포인트가 무엇인지

즉, 작업 후 답변도 다시 정리하지 않아도 되도록
보고 형식 자체를 지시문 안에서 강제합니다.

#### 작업 후 보고 시 필수 포함 항목
- 수정 완료 여부
- 테스트 실행 여부
- 실행한 테스트 항목
- 테스트 결과
- 미실행 테스트 항목
- 확인 불가 항목
- 남아 있는 위험 요소

## 5. 새 대화창 시작 프롬프트 작성 기준

새 대화창 시작 프롬프트는 항상 누락 방지용 인수인계 문서처럼 작성합니다.

반드시 아래 내용을 포함합니다.

### 5-1. 이 대화창에서 실제로 진행한 세부 작업
- 오늘 이 대화창에서 실제로 무엇을 했는지
- 어떤 순서로 확인 / 수정 / 검증했는지
- 어떤 테스트 데이터를 지웠는지
- 어떤 계정으로 어떤 화면을 확인했는지
- 어떤 문제를 발견했고 어떻게 정리했는지

### 5-2. 이전까지 누적 확정된 구조
- 역할 체계
- status 체계
- 승인 단계 구조
- delegation 구조
- 사용자별 권한 구조
- 최근 확정된 코드 / 배포 기준
- 반드시 유지해야 하는 구조

즉, 새창에서 맥락이 사라져도
프로젝트의 누적 확정 구조를 다시 복원할 수 있어야 합니다.

### 5-3. 오늘 새창에서 실제로 수정 / 검증 / 정리한 내용
새창 기준으로는 오늘 작업이므로,
해당 시점에서 실제 진행한 내용을 구체적으로 적습니다.

예:
- PR 생성 / 머지 / 배포
- conflict 해결 파일
- Firebase Console 데이터 정리
- 역할 원복
- 브랜치 삭제
- 로컬 정리
- 로그인 테스트 결과

### 5-4. 다음에 바로 이어갈 작업
반드시 다음 작업을 끊기지 않게 적습니다.
- 다음에 가장 먼저 확인할 것
- 아직 확정되지 않은 값
- 아직 안 끝난 정리 작업
- 다음 브랜치 작업 시작 조건
- 다음 수정 시 주의할 포인트

### 5-5. 최대한 자세하게 작성
새창 프롬프트는 요약문이 아니라
실제 이어받는 작업자가 바로 상황을 파악할 수 있는 상세 인계문으로 작성합니다.

짧게 줄이지 말고,
가능하면 아래까지 적습니다.
- 실제 문서 ID / PR 번호 / 머지 커밋
- 어떤 파일 버전을 채택했는지
- 어떤 테스트 데이터가 삭제되었는지
- 어떤 원복이 완료되었는지
- 아직 미확정인 항목이 무엇인지

## 6. tomorrow-checklist 문서 작성 기준

### 6-1. 파일명 규칙
클로드 알바 폴더 저장용 md 파일명은 반드시 아래 형식을 사용합니다.

`tomorrow-checklist-YYYY-MM-DD.md`

예:
- `tomorrow-checklist-2026-03-28.md`

### 6-2. 문서 내용 작성 기준
문서 안에서는 내일 이어갈 내용을 오늘 작업 기준으로 적습니다.

즉,
- 지금 시점의 오늘 작업은
- 새 창 / 내일 기준에서는 이전 작업이 됩니다.

따라서 문서 제목 / 구성은 아래처럼 맞춥니다.

예시 구조:
- 오늘 작업 기준
- 직전 작업에서 확인 완료된 핵심 결론
- 오늘 실제로 수정 / 검증 / 정리한 내용
- 오늘 작업 후 남은 이슈
- 다음으로 바로 이어갈 작업
- 주의할 민감 파일 / 금지 영역
- 회귀 체크 포인트
- 테스트 실행 여부 / 미실행 항목 / 확인 불가 항목

중요:
내일 열었을 때 바로 이어가야 하므로,
“내일 할 일” 식으로 뭉뚱그리지 말고
오늘 작업을 마친 시점에서 무엇이 남아 있는지 기준으로 적습니다.

## 7. 사실성 / 추정 금지 원칙

프로젝트 관련 지시문이나 새 대화창 시작 프롬프트를 작성할 때는,
반드시 현재 대화에서 사용자가 직접 제공한 정보와 현재 대화에서 실제로 확인된 내용만 기준으로 작성합니다.

확인되지 않은 브랜치명, 배포 상태, PR 번호, 파일 수정 내역, 테스트 결과를 임의로 추정해서 단정적으로 쓰지 않습니다.

새창 프롬프트나 인수인계 문서를 작성할 때는,
반드시 아래를 구분해서 적습니다.
- 이번 대화에서 실제로 진행한 작업
- 이전까지 누적 확정된 구조

정보가 없거나 현재 대화에서 확인되지 않은 항목은 추정하지 말고 아래 표현을 사용합니다.
- 미확인
- 사용자 제공 기준
- 직전 대화 기준

---

## 8. 항상 우선하는 작성 방향
아래 상황에서는 무조건 자세하게 씁니다.
- 브랜치가 여러 번 갈렸을 때
- hotfix / fix 브랜치가 섞였을 때
- Firebase Console 데이터 수정이 있었을 때
- role / roleGroup / status 체계가 함께 얽혀 있을 때
- 새창으로 이어갈 가능성이 높을 때
- 테스트 데이터 삭제 / 원복 작업이 포함될 때
- 한 번 잘못 건드리면 정상 기능이 깨질 가능성이 있을 때
- 수정 후 버그 / 오류 / 회귀 테스트 결과까지 남겨야 할 때

---

## 9. 금지할 작성 방식
다음 방식은 피합니다.
- 짧은 요약만 던지는 형식
- 현재 상태 설명 없이 바로 수정 지시만 하는 형식
- 금지 파일 / 금지 구조가 없는 지시문
- 필요한 부분 수정처럼 모호한 표현
- 알아서 판단 비중이 큰 지시문
- 회귀 체크 없는 지시문
- 테스트 여부가 빠진 지시문
- 작업 후 보고 형식이 없는 지시문
- 새창에서 맥락이 끊길 수 있는 축약형 프롬프트

---

## 10. 권장 출력 템플릿

아래 템플릿 순서를 기본으로 사용합니다.

1. 프로젝트명
2. 현재 상태
3. 이번 작업에서 최종적으로 확인된 핵심
4. 현재 누적 확정 구조
5. 이번 작업 목적
6. 우선순위
7. 건드리면 안 되는 로직 / 파일 / 구조
8. 수정 대상 파일과 수정 방식
9. 실제 작업 지시
10. 회귀 체크 포인트
11. 테스트 / 검증 항목
12. 절대 하지 말아야 할 것
13. 작업 후 보고 형식

새 대화창 시작 프롬프트는 아래 순서를 기본으로 사용합니다.

1. 프로젝트명
2. 이전까지 누적 확정된 구조
3. 직전 대화창 / 오늘 작업에서 실제로 진행한 세부 작업
4. 실제 수정 / 검증 / 정리 완료 내용
5. 아직 남은 작업
6. 가장 먼저 확인할 항목
7. 민감 파일 / 금지 영역
8. 다음 작업 요청 방식
9. 테스트 / 검증 주의사항
10. 작업 후 보고 방식

## 11. 최종 운영 규칙
앞으로 클로드 지시문 또는 새창 시작 프롬프트 요청 시,
항상 이 문서를 기준으로 작성합니다.

핵심 원칙은 다음과 같습니다.

- 존댓말 업무용 톤으로 작성
- 짧게 줄이지 않고 길고 자세하게 작성
- 현재 상태와 완료 작업 먼저 정리
- 이번 작업 목적과 우선순위 명확히 기재
- 건드리면 안 되는 로직 / 파일 / 구조 분리 명시
- 수정 파일과 수정 방식 구체화
- 회귀 체크 포인트 포함
- 코드 / 데이터 / 설정 변경 후에는 반드시 버그 / 오류 / 회귀 테스트 여부까지 보고
- 작업 후 보고 형식까지 지정
- 새창 프롬프트에는 실제 진행 내용과 누적 구조, 다음 작업까지 최대한 자세히 포함
- tomorrow-checklist 파일명은 반드시 `tomorrow-checklist-YYYY-MM-DD.md`
- 문서 안에서는 내일 이어갈 내용을 오늘 작업 기준으로 작성
