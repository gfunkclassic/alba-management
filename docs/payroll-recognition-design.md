## 1. 현재 배경

### 1.1 fmj-worklog 남양주 2026-04 파일 검증 완료
- 남양주 직원 9명(LM001~LM009) 2026-04 근태 엑셀 read-only 검증 완료
- 사용자가 양식 수정 요청 → 앱담당자가 외부 시스템 export 양식 정리
- 최신 수정본은 **실제 근태 사실 기준**으로 정리됨 (`2026-05-13 10:18` 기준)

### 1.2 최신 fmj-worklog 엑셀 표기 규칙
- **토요일/일요일**: 출근/퇴근/총근무/휴게/실제근무 모두 `-`, 상태·비고 공란
- **연차일**: 출퇴근/근무시간 모두 `-`, 상태 `연차`, 비고 공란
- **반차일**: 실제 출퇴근 기준 근무시간만 표시 (예: 출 08:53 / 퇴 13:00 → 실 4h), 상태 `오전 반차` 또는 `오후 반차`
- **정상근무일**: 실제 출퇴근 시간 + 점심 1h 차감 + 실제근무
- **하단 합계**: 실제근무 기준 합계 (연차/반차 인정시간 미포함)
- 잔존 제거된 표현: `주휴수당`, `주휴`, `토요일 주휴`, `일요일 휴무`, `월차`, `기본근로적용`, `반차 인정` (모두 unique 검색 0건 확인)

### 1.3 검증된 합계 (남양주 9명, 2026-04)
- 남양주 샘플에서 출퇴근 패턴 08:55~17:03, 점심 1h 차감 → **샘플 직원들의 1일 실근무 = 7h로 확인됨**
- 이는 해당 샘플 직원들의 계약/근무 패턴 예시값으로만 본다 — 사업장 단위 고정값으로 일반화하지 않음
- 일반 직원 합계(연차 없음, 본 샘플): 22 평일 × 7h = **154h**
- 토요일/일요일 미포함, 연차/반차 인정시간 미포함

## 2. 기본 원칙

| 원칙 | 의미 |
|---|---|
| **fmj-worklog 엑셀 = 실제 근태 사실 원본** | 외부 시스템에서 export된 출퇴근 기록을 신뢰의 원천(source of truth)으로 본다. 가상 시간 / 자동 보정 표기 금지. |
| **앱 = 급여 인정시간 자동 계산** | 연차/반차 인정시간, 주휴수당, 분 단위 절삭은 모두 앱 `calculateMonthlyWage` 에서 reason 기반으로 파생 계산한다. |
| **실제근무시간과 유급 인정시간을 섞지 않는다** | attendance 도큐먼트에는 실제 출퇴근만 저장한다. 인정시간은 계산 시점에만 파생값으로 산출한다. 노무 감사 추적성 확보. |
| **주휴수당은 엑셀 행에서 가져오지 않는다** | 토요일 행에 시간을 넣는 방식은 사용 안 함. 앱이 주별 합산 후 별도 산정한다. |
| **결근만 주휴 제외 사유** | 조기퇴근, 지각, 외출은 주휴 제외 사유 아님. 결근(`reason.includes('결근')`)만 박탈. |
| **1일 인정시간은 직원별 개별 계약 기준** | 사업장 단위 고정값(예: "남양주는 무조건 7h", "본사는 무조건 8h")으로 일반화하지 않음. 연차/반차 인정시간은 **해당 직원의 1일 계약 인정시간** 으로 계산. 남양주 2026-04 샘플에서 직원들의 계약 패턴이 7h로 일치했던 것은 예시일 뿐. |
| **1차 원천: `일 근무(h)` 입력 필드 (`employees.workHours`)** | 신규 인원 등록 화면 "근무 정보" 섹션의 `일 근무(h)` 가 이미 직원별로 `employees` 컬렉션에 저장됨. 2026-05-13 audit 결과(`scripts/audit-workhours.mjs`) 재직자 16명 누락 0건 / LM 9명 전원 7h 정합 / 출퇴근 정합성 불일치 0건 확인 → **1차 원천 확정**. 별도 필드 신설 없이 그대로 사용. 단 문자열 타입 2건은 계산 시 `Number()` 방어 처리 + 추후 데이터 정리 후보. |

## 3. 급여 인정시간 계산식

### 3.1 변수 정의
- `contractDailyHours`: **직원별 1일 계약 인정시간** (개별 계약 근무시간 기준). 사업장 단위 고정값 아님.
  - **1차 원천 확정**: Firestore **`employees.workHours`** 필드 (저장 키 확인됨; 라벨 `일 근무(h)`, UserFormModal.jsx:223; 컬렉션 `employees`, useEmployees.js)
  - 2026-05-13 audit (`scripts/audit-workhours.mjs`) 결과 — 재직자 16명 기준 누락 0건 / 0값 0건 / checkIn~checkOut 정합성 불일치 0건. LM 9명 전원 7h 정합. 별도 필드 신설 없이 1차 원천으로 채택 결정.
  - **방어 처리 필요**: 값 타입이 숫자/문자열 혼재 가능 (audit 결과 문자열 2건 확인). 계산 진입 시 `Number(employees.workHours)` 또는 동등 변환 + NaN 가드 필수.
  - **장기 후보 (당장 미도입)**: `contract_daily_minutes` 또는 `contract_daily_hours` 명시 필드 — 향후 의미 분리 필요 시 검토 (§11.2)
- `actualHours[d]`: d일의 실제 출퇴근 기준 근무시간 (점심 차감 후)
- `reason[d]`: d일의 상태 (`'정상' | '연차' | '오전 반차' | '오후 반차' | '조기퇴근' | '결근' | ''`)

### 3.2 산식
```
실근무 합계 = Σ actualHours[d]    (모든 d에 대해, 연차일은 0)

연차 인정시간 = count(reason[d] === '연차') × 해당 직원의 contractDailyHours

반차 인정시간 = Σ max(0, 해당 직원의 contractDailyHours - actualHours[d])
              for d where reason[d] ∈ {'오전 반차', '오후 반차'}

급여 인정 합계 = 실근무 합계 + 연차 인정시간 + 반차 인정시간
```

### 3.3 주요 특징
- 연차일은 출퇴근 미기록 → `actualHours = 0` 자연 처리
- 반차일은 실 출근 + 보전 = 해당 직원의 `contractDailyHours` 로 자동 cap (초과 방지)
- 야근(overtime) 은 별도 산정 (`overtime × wage × 1.5`), 본 산식과 분리
- 분 단위 절삭은 §6 별도 정책

## 5. 반차 정책

| 항목 | 정책 |
|---|---|
| **연차 차감** | 0.5일 (leave_balance.used_days += 0.5) — 기존 흐름 유지 |
| **급여 인정시간 산식** | `max(0, contractDailyHours - actualHours)` (해당 직원의 계약 기준 시간 사용) |
| **상한 (cap)** | 해당 직원의 1일 계약 인정시간을 초과하지 않음 (`max(0, ...)` 가 자동 보장) |
| **고정 가산 금지** | `contractDailyHours / 2` 고정(예: 3.5h) 가산 방식 사용 안 함 |
| **이유** | 실 출근 4h + 고정 3.5h = 7.5h 같은 계약 기준 초과 인정을 피함. 실 출근에 따라 자동 조정. |

### 5.1 케이스별 예시 (`contractDailyHours = 7h` 인 직원 가정)
| 실 출근 | 산식 | 인정시간 | 일별 합계 |
|---|---|---|---|
| 0h (출근 안 함) | max(0, 7-0) | 7h | 7h |
| 2h | max(0, 7-2) | 5h | 7h |
| 4h (LM009 04/29) | max(0, 7-4) | 3h | 7h |
| 6h | max(0, 7-6) | 1h | 7h |
| 7h+ (조기퇴근 아닌 정상) | max(0, 7-7) | 0h | 7h+ |

> 위 표는 계약 기준 7h 직원 가정. 8h 계약 직원이라면 동일 산식에 `contractDailyHours = 8h` 대입 — 예: 실 4h → 인정 4h → 일별 8h.

## 6. 분 단위 절삭 정책

| 항목 | 정책 |
|---|---|
| **반올림 금지** | `Math.round` 사용 금지. 회사 정책 명문 "반올림하지 않고 절삭". |
| **일별 절삭 금지** | 일별 `Math.floor` 적용 시 직원별 최대 -29h/월 손실 가능 → 직원 친화적이지 않음. |
| **월별 절삭 적용** | `Math.floor(monthly_paid_recognized_hours)` — 월 합산 후 1시간 미만 절삭 |
| **적용 위치** | `calculateMonthlyWage` 마지막, `actualBaseOvertimePay` 산출 직후 (잠정) |
| **야근 절삭** | 별도 검토 항목 (§11) — 동일 월 절삭 적용 vs 별도 절삭 미정 |
| **주휴 weeklyHours 절삭** | 별도 검토 항목 (§11) — 절삭 전/후 어떤 값 사용할지 미정 |

### 6.1 적용 위치 후보
- (a) 일별 절삭: `calculateDailyWage` 결과 `Math.floor(totalHours)` ✕ 직원 손실 큼
- (b) 주별 절삭: weeklyHours 합산 직후 `Math.floor` △ 중간
- (c) **월별 절삭: 월 인정 합계 산출 후 `Math.floor`** ✅ 권장

### 6.2 시행 영향
- 도입 시점 이전 정산 결과와 차이 발생 → §11.4 시행월 명문화 + 직원 사전 고지 필수

## 7. 주휴수당 정책

| 항목 | 정책 |
|---|---|
| **엑셀 행에서 가져오지 않음** | 토요일 행 시간 0 (현 fmj-worklog 수정본 정합). |
| **앱 별도 산정** | `calculateMonthlyWage:365-379` 기존 로직 그대로 유지. `min(weeklyHours / 5, 8) × wage`. |
| **주 귀속** | 해당 주 목요일이 속한 달 (ISO 8601 방식). 월말 이월 자동 처리. |
| **결근 → 주휴 박탈** | `reason.includes('결근')` 키워드 매칭 (기존 동작). |
| **조기퇴근 → 주휴 박탈 안 함** | 결근 아님. 주 15h 이상이면 주휴 정상 부여. |
| **연차/반차 인정시간을 weeklyHours에 포함?** | **§11.5 미확정** — 포함하지 않으면 연차 사용 주 주휴 누락 위험. 포함하면 정합. 사용자 정책 확정 필요. |

### 7.1 토요일 제거 후 중복 위험
- ✅ **해소** — 토요일 attendance가 비어 있으면 `weeklyHours` 에 0h 합산. 별도 주휴 산정과 충돌 없음.
