# DEV_CHANGE_GUIDE.md
## Development Change Guide (Regression Prevention)

### 목적
기능 수정 또는 추가 시 **요청된 범위만 변경**하고  
기존에 정상 동작하던 기능이 깨지는 것을 방지한다.

모든 개발자는 이 가이드를 따라야 한다.

---

# 1. 최우선 원칙

1. Scope Lock  
요청된 기능 범위 밖의 코드는 수정하지 않는다.

2. No Refactoring Unless Requested  
리팩터링 / 구조 변경 / 코드 정리는 요청이 있을 때만 수행한다.

3. Minimal Diff  
변경 라인은 최소화한다.

4. Backward Compatibility  
기존 데이터 및 기능과 호환성을 유지한다.

5. Regression Prevention  
기존 기능이 깨지지 않았는지 반드시 확인한다.

---

# 2. 작업 프로세스

## Step 1. 변경 요청 분석
다음 내용을 명확히 한다.

- 변경 목적
- 변경 기능
- 변경 범위
- 변경 금지 영역
- 완료 기준

---

## Step 2. 영향도 분석

수정 전 반드시 다음을 확인한다.

- 영향을 받을 수 있는 기능
- 영향을 받을 수 있는 파일
- 회귀 가능성

---

## Step 3. 구현

다음 원칙을 따른다.

- 요청된 기능만 수정
- 기존 로직 변경 최소화
- 불필요한 코드 수정 금지

---

## Step 4. 회귀 테스트

다음 기능이 정상 동작하는지 확인한다.

### 기본 기능
- 로그인
- 권한 체크

### 연차 기능
- 연차 신청
- 연차 취소
- 팀 승인
- 최종 승인

### 데이터
- leave_requests 정상 기록
- leave_balance 정상 차감
- approvals 로그 생성
- notifications 정상 생성

---

# 3. 변경 금지 영역

아래 영역은 요청 없이는 수정 금지

- Firestore Rules
- Cloud Functions 승인 로직
- 인증 / 권한 처리
- 데이터 스키마
- 승인 트랜잭션 로직

---

# 4. Pull Request 규칙

PR에는 다음 정보가 반드시 포함되어야 한다.

### 변경 내용
무엇을 수정했는지 설명

### 변경 이유
왜 수정했는지 설명

### 변경 범위
수정된 파일 목록

### 테스트 결과
회귀 테스트 결과

### 리스크
발생 가능한 문제

---

# 5. 배포 규칙

배포 전 반드시 확인한다.

- 기존 기능 정상 동작
- 데이터 무결성 유지
- 승인/차감 로직 정상

---

# 6. AI 개발 에이전트 규칙

Claude / Gemini / Antigravity 등  
AI 개발 에이전트는 다음 규칙을 반드시 따른다.

1. 요청된 기능만 수정
2. 기존 코드 수정 최소화
3. 리팩터링 금지
4. 기존 기능 변경 금지
5. 변경 영향 분석 후 수정

---

# 7. Rollback 정책

문제 발생 시

1. 이전 버전으로 즉시 롤백
2. 데이터 변경 확인
3. 수정 후 재배포
