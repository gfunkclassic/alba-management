# DEV CHANGE GUIDE
안전한 코드 수정 규칙 (Regression 방지)

이 문서는 코드 수정 시 기존 기능이 깨지는 문제를 방지하기 위한 공통 규칙입니다.
개발자와 AI(Claude, Gemini, Copilot)는 아래 규칙을 반드시 따릅니다.

---

# 1. 수정 기본 원칙

코드 수정 시 반드시 지켜야 할 원칙

1. 요청된 기능만 수정한다
2. 기존에 문제 없던 코드는 변경하지 않는다
3. 변경은 최소한으로 한다
4. 포맷팅/정렬 변경 금지
5. 리팩터링은 별도 PR로 진행

즉

"필요한 코드만 최소한으로 수정"

---

# 2. 금지 사항

다음 작업은 금지합니다.

- 파일 전체 포맷 변경
- 사용하지 않는 코드 정리
- import 정리
- 파일 이름 변경
- 폴더 구조 변경
- 대규모 리팩터링

이런 작업은 반드시 별도 PR에서 진행합니다.

---

# 3. 코드 수정 절차

코드 수정은 다음 순서로 진행합니다.

1️⃣ 브랜치 생성

예시
fix/leave-bug  
feature/add-notification  

2️⃣ 코드 수정

- 수정 대상만 변경
- 최소 변경 유지

3️⃣ PR 생성

PR에는 반드시 아래 내용을 작성합니다

변경 내용  
변경 이유  
변경 파일  

4️⃣ CI 확인

아래 두 CI가 반드시 성공해야 합니다

CI / Web build (Vite)  
CI / Functions build (TypeScript)

5️⃣ Merge

CI 성공 후 Merge 진행

---

# 4. PR 체크리스트

PR 작성 시 아래 항목 확인

[ ] 요청한 기능만 수정  
[ ] 기존 기능 변경 없음  
[ ] 불필요한 코드 수정 없음  
[ ] CI 통과  

---

# 5. AI 코드 수정 규칙

Claude / Gemini / Copilot 에게 코드 수정 요청 시 다음 규칙을 반드시 따릅니다.

요청된 기능만 수정하세요.

기존에 문제 없던 코드는 변경하지 마세요.

포맷팅 변경, 리팩터링, 파일 이동은 금지합니다.

변경은 최소한의 코드만 수정하세요.

수정 후 아래 내용을 반드시 보고하세요.

- 수정한 파일
- 수정한 함수
- 기존 기능 영향 여부

---

# 6. 문제 발생 시 롤백

문제 발생 시 가장 빠른 해결 방법

PR 페이지 → Revert 버튼 사용

또는

git revert <commit>

---

# 핵심 원칙

"요청한 코드만 최소한으로 수정하고
기존 기능은 절대 건드리지 않는다."
