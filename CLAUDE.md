# 아르바이트 급여·연차 관리 프로그램 작업 규칙

## 기본 원칙
- 현재 대화와 저장소에서 실제 확인된 내용만 기준으로 작업합니다.
- 확인되지 않은 브랜치명, PR 번호, 배포 상태, 수정 파일, 테스트 결과는 절대 추정하지 않고 `미확인`으로 표기합니다.
- 기존에 정상 동작 중인 기능은 불필요하게 건드리지 않습니다.
- 작은 수정이라도 회귀 위험을 반드시 함께 보고합니다.
- 작업 전 현재 상태와 완료된 작업을 먼저 정리합니다.
- 작업 목적과 우선순위를 명확히 적습니다.
- 건드리면 안 되는 파일, 로직, 구조를 분리해서 명시합니다.
- 수정 파일과 수정 방식을 구체적으로 적습니다.
- 작업 후 보고 형식을 반드시 맞춥니다.

## 민감 영역
아래 영역은 특별한 명시가 없으면 건드리지 않습니다.

- 승인 단계 구조 전체
- 권한 / roleGroup / delegation 관련 로직
- Cloud Functions 차감 로직
- 취소 로직
- 인증 / 권한 체크
- 저장 구조:
  - applied_dates
  - start_date
  - end_date
  - day_count

## 작업 방식
- 가능한 한 최소 파일, 최소 범위로 수정합니다.
- 현재 요청 범위 밖의 리팩토링은 하지 않습니다.
- 표시(UI) 수정과 저장/승인 로직 수정은 구분해서 다룹니다.
- 프론트에서 차단하는 경우에도 필요하면 제출 직전 재검증을 유지합니다.
- 구 데이터 fallback이 있으면 제거하지 말고 유지 여부를 먼저 확인합니다.

## PR 본문 형식
PR Guard 통과를 위해 PR 본문은 아래 구조를 기본으로 사용합니다.

- 변경 내용
- 변경 이유
- 변경 범위
- 롤백

필요하면 추가로 아래를 넣습니다.
- 확인 사항
- 수동 확인 포인트

## handoff / 체크리스트 규칙
- tomorrow-checklist 파일명은 반드시 아래 형식을 사용합니다.
  - `tomorrow-checklist-YYYY-MM-DD.md`
- 문서 내용은 문서 작성 시점의 “오늘 작업” 기준으로 적습니다.
- 새 창/내일 기준으로 표현이 꼬이지 않도록 주의합니다.
- 추정하지 말고, 확인되지 않은 항목은 반드시 `미확인`으로 적습니다.

## 이 프로젝트에서 최근 중요 원칙
- FULL 연차는 개별 날짜 누적 선택 방식이 기본입니다.
- HALF_AM / HALF_PM은 단일 날짜 선택 방식 유지가 기본입니다.
- 주말 / 공휴일은 선택 불가 원칙입니다.
- 연차 신청 화면의 표시 개선과 저장 구조 변경은 구분해서 판단합니다.
- 잔여 연차 초과 선택/신청은 허용하면 안 됩니다.
- `남은 연차 / 산정 일수`는 사용자에게 즉시 이해되게 보여주는 방향을 우선합니다.

## Claude 응답 톤
- 존댓말의 업무용 톤으로 작성합니다.
- 짧게 퉁치지 말고, 실제 복붙해서 바로 쓸 수 있게 씁니다.
- 사용자가 “클로드용 지시문”, “새 대화창 시작 프롬프트”, “인수인계 문서”를 요청하면 누락 방지 중심으로 길고 자세하게 작성합니다.
